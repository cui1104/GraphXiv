# Plan 07-02.5 — Metric & Ground-Truth Overhaul

## Why this plan exists

After completing 07-02 (600-row benchmark CSV on RunPod RTX 4090, all 4 conditions at 150/150 success), analysis showed that the current metrics are **systematically biased toward GROBID** and fail to exercise features the router, MinerU, and Docling are actually built for.

### Concrete findings from the 07-02 CSV (2026-04-17)

| cond    | n   | sec_med | match_mean | heading_count_parser_med |
|---------|-----|---------|------------|--------------------------|
| mineru  | 150 | 13.7s   | 0.47       | 26                       |
| grobid  | 150 | 2.4s    | 0.66       | 20                       |
| docling | 150 | 14.3s   | 0.48       | 28                       |
| router  | 150 | 11.6s   | 0.59       | 23                       |

GT median heading count: 14.

### Three problems identified

1. **`heading_match_rate` is precision only** (matched / parser_headings). A parser that emits fewer headings gets a higher score — GROBID wins by under-extracting. No recall, no F1.
2. **No content-richness metrics.** MinerU/Docling are DL models built for figures, equations, tables, reading order. Current metrics score none of that.
3. **Router's differentiator is not exercised.** Plan 03-04 / 04-CONTEXT.md D-02 state the router's win is **rule-based dot-count hierarchy reconstruction across whichever child parser it picks** (MinerU or GROBID). But:
   - `benchmark/run_benchmark.py:113` — `run_mineru_standalone` drops `sec_num` and emits only `{heading, text}`.
   - `benchmark/run_benchmark.py:235` — `run_router_standalone` passes through child sections unchanged. No dot-counting. No normalization step.
   - Heading matching is on flat strings; hierarchy-depth is never scored.

## Goal

Produce a revised 600-row `benchmark.csv` with an expanded metric set that fairly measures:
- **Heading quality** via precision, recall, and F1 (not precision alone)
- **Hierarchy reconstruction** via depth-aware matching (the router's claimed win)
- **Content richness** via body-token volume, figure count, formula count, reference count

And a revised GT schema v2 that includes the new ground-truth signals.

## Scope boundaries

- **In scope:** metric functions, parser output extensions, router normalization step, GT re-extraction, CSV regeneration, unit tests.
- **Out of scope:** changing the corpus sample (150 papers stay), changing the 4 conditions (mineru/grobid/docling/router), re-running Phase 4 normalize on the production DB.

## Tasks

### Task 1 — Extend parser outputs to preserve `sec_num` and structural counts

File: `benchmark/run_benchmark.py`

- `run_mineru_standalone` — emit `sections = [{heading, sec_num, text}]` by reading `item.get("sec_num")` from MinerU `content_list`. Also count `type==equation`, `type==image`, `type==table` items into `{formula_count, figure_count, table_count}`. Return `(sections, content_list, struct_counts)`.
- `run_grobid_standalone` — extend `_parse_tei_fulltext_sections` (or inline in benchmark) to capture `<div n="...">` attribute into `sec_num`. Count `<formula>`, `<figure>`, `<listBibl>/<biblStruct>` from the TEI. Return `(sections, tei_bytes, struct_counts)`.
- `run_docling_standalone` — read `item.level` on `section_header` items to derive `sec_num`-equivalent (depth-only; Docling does not emit numeric hierarchy strings). Count `doc.pictures`, `doc.tables`, formula-label items into struct_counts. Return `(sections, tables, doc, struct_counts)`.

### Task 2 — Router hierarchy builder

File: `benchmark/run_benchmark.py`

Add `_apply_dot_count_hierarchy(sections: list[dict]) -> list[dict]` that:
- Takes raw sections with `sec_num` strings.
- Computes `depth = sec_num.count(".") + 1` (or `depth = None` if no sec_num).
- Computes `parent_sec_num` by walking backward from the current section and finding the largest-depth sec_num where `parent_sec_num.count(".") == depth - 2` and the current sec_num starts with `parent_sec_num + "."`.
- Emits `sections = [{heading, sec_num, depth, parent_sec_num, text}]`.

Wire into `run_router_standalone`: after the child parser returns, run sections through `_apply_dot_count_hierarchy` before returning. **This is the only condition that applies the builder** — the point is to show the router's unique win.

### Task 3 — New metric functions

File: `benchmark/metrics.py`

Add:
- `compute_heading_precision_recall_f1(parser_headings, gt_headings) -> (p, r, f1)` — uses existing `heading_matched` logic but produces all three from matched/|parser|/|gt|.
- `compute_hierarchy_f1(parser_sections, gt_sections) -> float` — for each (heading, depth) pair, treat GT as ground-truth set; compute F1 of the set `{(normalized_heading, depth)}`. Requires GT with sec_num (Task 4).
- `body_token_count(sections) -> int` — sum tokens across `section.text` using `tiktoken` cl100k_base (consistent with `app/tasks/normalize.py`).
- `count_figures(struct_counts) -> int`, `count_formulas(...)`, `count_references(...)` — identity passthroughs; exist for consistency and testability.

Keep existing `heading_matched`, `coherent_section_pct`, `_table_completeness_score` unchanged — they remain in the new CSV as orthogonal metrics.

## Expected outcomes (hypotheses to test)

1. GROBID's heading **precision** stays highest; its **recall** is lowest → its **F1** drops below MinerU/Docling.
2. MinerU/Docling have **higher figure_count and formula_count** recall than GROBID (near-zero for GROBID).
3. Router's **hierarchy_f1** is strictly higher than all standalone conditions.
4. GROBID's `reference_count` is highest (dedicated bibliography model).

If any hypothesis fails, that is itself a FINDINGS-worthy result. No hypothesis is load-bearing for the plan.

## Risks

- **GT re-extraction nondeterminism** — Opus may return slightly different sec_num strings per run. Mitigate: temperature=0, seed if available, cache by paper_id.
- **Docling depth fidelity** — Docling emits `level` (int), not numbered sec_num. Accept that Docling's `hierarchy_f1` will be structurally penalized. Note in FINDINGS.
- **pymupdf `page.find_tables()` false positives** — already used in router; unchanged here.
